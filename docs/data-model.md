# Helm V1 — Data model

> Drizzle schema in [`db/schema/`](../db/schema) is the source of truth.
> This doc explains the *why* — column-level definitions live in the `.ts`
> files.

## Carried forward from the legacy app

V1 maintains parity with the legacy CSV schema in `old_database/` so the
historical dataset imports without re-keying. UUIDs and `invoice_number`
values are preserved verbatim by `scripts/import-legacy.ts`.

| Legacy table | V1 table | Notable changes |
|---|---|---|
| `clients` | `clients` | unchanged |
| `invoices` | `invoices` | PK column renamed `invoice_id` → `id` |
| `invoice_line_items` | `invoice_line_items` | PK renamed `line_id` → `id` |
| `payments_received` | `payments_received` | PK renamed `payment_id` → `id` |
| `time_entries` | `time_entries` | PK renamed `entry_id` → `id`; **`description` dropped** per V1 spec |
| `tax_ledger` | `tax_ledger` | PK renamed `tax_id` → `id` |
| `tax_payments` | `tax_payments` | PK renamed `payment_id` → `id` |
| `invoice_tax_links` | `invoice_tax_links` | PK renamed `link_id` → `id` |
| `transfers` | `transfers` | PK renamed `transfer_id` → `id` |
| `settings` | `settings` | unchanged (key/value) |

## Relationships at a glance

```
clients ──┬─< invoices ─┬─< invoice_line_items
          │             ├─< payments_received
          │             └─< invoice_tax_links >─ tax_payments >─ tax_ledger
          └─< time_entries ─> invoices  (nullable, ON DELETE SET NULL)

transfers ─> tax_ledger  (company link, nullable)
transfers ─> tax_ledger  (personal link, nullable)

settings  (key/value, no FKs)
```

## Conventions

- **PKs:** UUID v4, generated by `gen_random_uuid()`. Legacy UUIDs preserved
  verbatim during the one-shot import.
- **Money:** `NUMERIC(15, 2)`. Tax rates: `NUMERIC(6, 4)`. Hours:
  `NUMERIC(5, 2)`.
- **Dates:** Postgres `DATE` for date-only fields (issue, due, work,
  payment). `TIMESTAMP WITH TIME ZONE` (UTC) for `created_at` /
  `updated_at`.
- **Statuses:** `VARCHAR(20)` rather than Postgres ENUMs in V1 — easier to
  evolve without `ALTER TYPE` migrations. May tighten with a check
  constraint or ENUM later.
- **Indexes:** all foreign-key columns are explicitly indexed (Postgres
  doesn't auto-index FKs). Plus targeted indexes on
  `invoices.status`, `invoices.issue_date`,
  `payments_received.payment_date`, `transfers.transfer_date`.
- **`updated_at`:** application sets it explicitly on UPDATE. If that
  becomes noisy or error-prone, add a Postgres trigger.

## Foreign-key delete behaviour

| Edge | On delete | Why |
|---|---|---|
| `invoice_line_items.invoice_id` → `invoices` | CASCADE | Lines belong to the invoice. |
| `time_entries.invoice_id` → `invoices` | SET NULL | Unlink, don't delete the work. |
| `invoice_tax_links.invoice_id` → `invoices` | CASCADE | Junction. |
| `invoice_tax_links.tax_payment_id` → `tax_payments` | CASCADE | Junction. |
| `payments_received.invoice_id` → `invoices` | NO ACTION (default) | Prevent silently deleting paid invoices. |
| `transfers.tax_ledger_link_*` → `tax_ledger` | NO ACTION | Soft links; manual cleanup. |
| `tax_payments.tax_id` → `tax_ledger` | NO ACTION | Soft link. |

## Migration semantics (legacy CSV → Postgres)

`scripts/import-legacy.ts` will:

1. Read each CSV in dependency order: `clients` → `invoices` →
   `invoice_line_items` → `payments_received` → `time_entries` →
   `tax_ledger` → `tax_payments` → `invoice_tax_links` → `transfers` →
   `settings`.
2. Map empty strings to `NULL` (legacy CSVs use `""` for absent values).
3. Coerce booleans (`"True"` / `"False"`) and numerics.
4. Drop `time_entries.description` on import.
5. Preserve UUIDs and `invoice_number` exactly. Use `INSERT … ON CONFLICT
   DO NOTHING` so the script is idempotent for repeated runs against dev.
6. After import, validate row counts and FK integrity.

## What's not modelled yet (V2)

To be added when we build the Personal side and the Receipts feature:

- **`categories`, `transactions`, `budgets`** — Personal, YNAB-style budgeting.
- **`holdings`, `holding_transactions`** — Personal, investment tracking.
- **`receipts`** — Business, image references in S3 + extracted fields.
- **`csv_imports`** — audit trail of bank-export ingestion.
- **`users`** — only if/when Helm becomes multi-user.

## Multi-tenancy

V1 is single-user. There is no `owner_id` on any table. Auth at the API
gate (Cognito JWT validation) is the only access boundary. Adding
multi-user support later requires a backfilled `owner_id` column on every
top-level table and an `owner_id = $current_user` filter on every query.
